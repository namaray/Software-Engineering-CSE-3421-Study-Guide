Design Patterns (GoF)

A standard solution to a common programming problem. It ensures loose coupling and increases understandability.

UML Legend (Crucial for Diagrams):
  • Solid Line + Hollow Triangle: Generalization (Inheritance/Extends)
  • Dashed Line + Hollow Triangle: Realization (Implements Interface)
  • Dashed Line + Open Arrow: Dependency ("Uses")
  • Solid Line + Diamond: Composition/Aggregation ("Has a")

1. Creational Patterns

Concerned with how objects are instantiated.

A. Factory Method

Definition Defines an interface for creating an object but lets subclasses decide which class to instantiate.
Motivation: Removes the instantiation of actual implementation classes from client code.
<<interface>>Shape
+ draw()
Circle
+ draw()
Rectangle
+ draw()

The Factory:

ShapeFactory
+ getShape(type): Shape

Consequences: Makes code more robust. You can add a "Triangle" later without breaking the Client code.

B. Abstract Factory

Definition Provide an interface for creating families of related or dependent objects (Factory of Factories).

Example: A VehicleFactory that produces unrelated parts (Wheels, Engine) that must fit together for a specific Car Model.

  • Client: Asks FactoryProducer for a factory.
  • AbstractFactory: Interface for getShape() and getColor().
  • Concrete Factories: PaintFactory (creates shapes/colors) vs AnimationFactory.

C. Singleton

Definition Ensures a class has only one instance and provides a global point of access to it.
DbConnection
- static instance: DbConnection
- DbConnection() (Private!)
+ static getInstance(): DbConnection
Client

Key Code concept: The constructor is private so no one else can say new DbConnection(). They must use getInstance().

2. Structural Patterns

Concerned with how classes and objects are composed (combined).

A. Adapter Pattern

Definition Converts the interface of a class into another interface the clients expect.
Analogy: A Travel Plug Adapter (US Plug into EU Socket).
Client
Adapter
+ request()
(calls adaptee.specificRequest)
Adaptee
+ specificRequest()

Practice Problem: Video Player. System plays mp4/vlc. Client wants to play mp3 (audio). Use an AudioAdapter to translate the request.

B. Decorator Pattern

Definition Attaches additional responsibilities to an object dynamically. An alternative to subclassing for extending functionality.
Principle: Open for Extension, Closed for Modification.

Structure: A Decorator is a Shape (Inheritance) AND has a Shape (Composition).

<<interface>>Shape
+ draw()
Circle
+ draw()
ShapeDecorator
- shape: Shape
+ draw() { shape.draw() }
RedBorderDecorator
+ draw() {
  super.draw();
  setRedBorder();
}

3. Behavioral Patterns

Concerned with communication between objects.

A. State Pattern

Definition Allows an object to alter its behavior when its internal state changes. The object will appear to change its class.

Example: Traffic Light (Red state vs Green state behavior) or TV Remote.

Context (Remote)
- state: State
+ pressButton() {
  state.doAction()
}
<<interface>>State
+ doAction()

B. Strategy Pattern

Definition Defines a family of algorithms, encapsulates each one, and makes them interchangeable at runtime.

Example: Sorting Algorithms or Payment Methods.

<<interface>>Strategy
+ execute(a, b)
AddStrategy
SubStrategy
MulStrategy

Practice Problem: Sorting an array. Client wants to find the 2nd largest number. You can swap between BubbleSort, QuickSort, etc., without changing the client code.
