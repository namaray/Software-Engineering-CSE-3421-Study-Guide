UI and UX Design

Course: CSE 3421 | Source: Lecture Slides

1. UI vs. UX

The "Face" of your software. If you don't take care of the appearance, users may not trust the functionality.

UI (User Interface): The visual elements (The ketchup bottle design, the cap, the label).

UX (User Experience): The feeling of using it (Is it hard to get the ketchup out? Do you have to hit the bottom?).

Usability

The effectiveness with which users can accomplish tasks. It is measured by:

  • Learnability: Is it easy to learn for the first time?
  • Efficiency: Once learned, is it fast to use?
  • Safety: Are errors few and recoverable?

Cost of bad UX: 50% of "malfunctioning" returns are actually just too hard to use. (Example: Three Mile Island nuclear meltdown was caused by ambiguous UI).

2. Design Process

The "User" Philosophy

  • You are not the user: Developers think differently than end-users. UI is about communicating with users, not other programmers.
  • The User is always right: If they make consistent mistakes, it is the system's fault, not theirs.

Steps in UI Design

  1. Planning: Understand customer values, brainstorm, make feature lists, plan navigation.
  2. Prototyping & Sketching:
    • Known as "Wireframe modeling" (Skeleton of the UI).
    • Can be Manual (Paper/Pencil) or Electronic (Figma, Adobe XD).
    • Benefits: Aids implementation, helps discover requirements/test cases, ensures user satisfaction early.
  3. Optimization: Refine the design.
  4. Implementation: Code the interface.

4. Shneiderman's 8 Golden Rules

Critical for the exam. These are the fundamental principles of good design.

1. Strive for Consistency:
Use identical terminology (prompts, menus) and consistent layouts throughout the app. Don't use "Save" on one screen and "Write" on another.
2. Shortcuts for Frequent Users:
As users get expert, they want speed. Use abbreviations, hotkeys (Ctrl+S), and macros.
3. Offer Informative Feedback:
For every action, the system should respond.
Minor action: Modest response (button click effect).
Major action: Substantial response (ProgressBar, "File Saved" modal).
4. Design Dialog to Yield Closure:
Actions should have a Beginning, Middle, and End. Give the user a sense of relief/accomplishment (e.g., "Installation Complete" screen).
5. Simple Error Handling:
Design so users cannot make serious errors (gray out invalid buttons). If an error happens, offer clear, simple mechanisms to fix it.
6. Permit Easy Reversal of Actions:
The "Undo" button relieves anxiety. Users will explore more if they know they can fix mistakes.
7. Internal Locus of Control:
Make the user feel like the "Boss". The user initiates actions; the system responds. Avoid automated surprises that take control away.
8. Reduce Short-term Memory Load:
Humans have limited short-term memory. Keep displays simple. Don't make users remember a code from Screen 1 to type into Screen 3.

5. UI Components (Widgets)

  • Buttons: For actions. Use Verbs (e.g., "Save," "Send"), not generic "OK/Yes".
  • Check Boxes: Independent options (On/Off). Multiple can be selected.
  • Radio Buttons: Related choices where only one can be selected at a time.
  • Text Fields: When users need to type free input.
  • Combo/List Boxes: Use for many fixed choices to save screen space.
  • Menus: For infrequent actions. (Note: Users often dislike digging through menus; provide toolbars for common tasks).
  • Tooltips: Hover text for help/information.
  • Tabbed Panes: Switching between screens without losing context.
