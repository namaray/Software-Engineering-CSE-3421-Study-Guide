Software Architecture

Course: CSE 3421 | Core Concept: Modularization & Distribution

1. N-Tier Architecture Overview

Definition: An approach where the project/program is distributed among N separate computers in a distributed network.

Why use it? To divide software into manageable pieces.

The 3 Logical Layers

  • Presentation Layer (UI): The top-most level. Translates tasks/results for the user (The Interface).
  • Business Logic Layer: The coordinator. Makes logical decisions, performs calculations, moves data.
  • Data Access Layer: Storage. Retrieving and saving info to the database/file system.

Impacts of Multi-Tiering

Splitting systems into tiers improves: Scalability, Reliability, Fault Tolerance, Security, Maintainability, Reusability.

2. Types of N-Tier Architecture

How we physically separate the code onto different machines.

A. 3-Tier Architecture (The Standard)

The logic is separated from the client and the database. This is the most common web architecture.

Client Tier (PC / Browser)
UI Only
↔
Application Tier (App Server)
Business Logic
↔
Database Tier (DB Server)
Data Storage

B. 2-Tier Architecture

The client communicates directly with the database. The business logic usually sits on the Client (Fat Client) or inside the Database (Stored Procedures).

Client Tier (Client Computer)
UI + Logic
↔
Database Tier (DB Server)
Data

C. 1-Tier Architecture

Everything runs on a single machine or a simple file server shared network.

1-Tier (All-in-One)
Client + Logic + Data
(e.g., Desktop App, File Server)

3. Architectural Patterns

MVC (Model-View-Controller)

A specific pattern for the Presentation/Logic separation.

Component Function
Model Data-related logic. Represents data transferring between View and Controller.
View UI logic. Text boxes, dropdowns, final output.
Controller Interface. Processes business logic, handles requests, manipulates Model.

Frameworks: Spring (Java), Django (Python), Laravel (PHP).

4. Monolithic vs. Microservices

Monolithic Microservices
Built as a single, indivisible unit. Collection of small, independent units.
Tight Coupling: Hard to change one part without affecting others. Low Coupling: Services talk via APIs (HTTP).
Shared Database. Each service has its own Database.
Pros: Low cost, Simple to develop initially. Pros: Scalable, Maintainable, Flexible technology.
Cons: Hard to scale, hard to maintain as it grows. Cons: Extra complexity, cross-cutting concerns.

5. Specialized Architectures

Micro-Kernel (Plug-in) Architecture

Structure: Core System (Minimal) + Plug-in Modules (Specialized).
  • Concept: Core keeps the system running; Plug-ins add features.
  • Requirement: Plugins need "handshaking code" to talk to the core.
  • Use Case: Applications that need frequent updates or dynamic rules (e.g., IDEs like VS Code).

Event-Driven Architecture

Structure: Decoupled event processing components.
  • Concept: Components asynchronously receive and process events.
  • Flow: Event -> Queue -> Mediator -> Channel -> Processor.
  • Pros: Agile, High performance, Easy debugging.
