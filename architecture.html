Software Architecture

Course: CSE 3421 | Concept: Modularization & Distribution

Overview

Architecture helps us modularize and distribute software into manageable pieces.

N-Tier Architecture: An approach where the project is distributed among N separate computers in a distributed network.

Why Multi-tier? It improves Scalability, Reliability, Fault Tolerance, Security, Maintainability, and Reusability.

1. Types of N-Tier Architecture

How the physical layers (Client, Server, Database) are distributed.

A. 3-Tier Architecture (Standard Web App)

Separates Presentation, Logic, and Data into three distinct layers/machines.

Client Tier UI / Browser
↔
Application Tier Business Logic
↔
Database Tier Data Storage

B. 2-Tier Architecture

Client talks directly to the Database. Logic is either on the Client (Fat Client) or in the DB (Stored Procedures).

Client Tier UI + Logic
↔
Database Tier Data Storage

C. 1-Tier Architecture

Everything runs on a single machine (e.g., Desktop app, File server based).

2. Architectural Design Patterns

Standard ways to structure the code and components of the software.

A. Layered Pattern: MVC

Model-View-Controller. A pattern used to separate the representation of information from the user's interaction with it.

The 3 Components
1. Model Data-related logic. Represents the data transferring between View and Controller.
2. View UI Logic. The visible parts (text boxes, dropdowns, final output).
3. Controller Interface. Processes business logic, handles requests, manipulates Model.
Frameworks Spring (Java), Django (Python), Rails (Ruby), Laravel (PHP). Advantages
  • Rapid development
  • Low coupling (Separation of concerns)
  • Efficient debugging
  • High maintainability

B. Monolithic Architecture

Built as a single, indivisible unit. All functions are managed and served in one place.

  • Pros: Low cost, simple to develop and deploy (initially).
  • Cons: Tight coupling, difficult to scale, difficult to maintain as it grows.

C. Microservice Architecture

Breaks the app down into a collection of smaller independent units.

Characteristics:
  • Each service has its own logic and own database.
  • Communication via lightweight mechanisms (HTTP APIs).
Comparison: Monolithic vs. Microservices
Feature Monolithic Microservices
Structure Single Unit Collection of Services
Database Shared Per Service
Coupling Tight Low

D. Micro-Kernel (Plug-in) Architecture

Consists of a Core System and several Plug-in Modules.

  • Core System: Minimal functionality to keep the system operational.
  • Plug-ins: Independent components with specialized processing.
  • Key Requirement: Plug-ins need "handshaking code" so the core recognizes them.

Use Case: IDEs (Eclipse, VS Code) where the core is the editor, and languages/tools are plugins.

E. Event-Driven Architecture

Made of decoupled, single-purpose components that asynchronously receive and process events.

  • Flow: Event -> Event Queue -> Event Mediator -> Event Channel -> Event Processors.
  • Pros: Agile, High performance, Easy debugging, Low coupling.
