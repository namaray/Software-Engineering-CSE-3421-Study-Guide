Software Architecture

Course: CSE 3421 | Topic: Modularization & Distribution

1. N-Tier (Layered) Architecture

An approach where the program is distributed among N separate computers in a distributed network. It helps modularize software into manageable pieces.

Impacts of Multi-Tiering:
  • Scalability (Easier to grow)
  • Reliability & Fault Tolerance
  • Security
  • Maintainability & Reusability

The 3 Main Tiers

1. Presentation Tier The User Interface (UI). Translates tasks and results into something the user can understand.
2. Logic Tier The "Brain". Coordinates commands, makes logical decisions, performs calculations, and moves data between layers.
3. Data Tier Storage. Information is stored and retrieved here (Database or File System).

2. MVC Pattern (Model-View-Controller)

One of the most common layered architectural patterns. Other variations include HMVC, MVVM, and MVP.

Component Function
Model Handles Data Logic. Represents the data being transferred between View and Controller.
View Handles UI Logic. The visible parts (text boxes, dropdowns, final output).
Controller The Interface. Processes business logic, handles incoming requests, manipulates the Model, and updates the View.

Common Frameworks: Spring (Java), Django (Python), Rails (Ruby), Laravel (PHP).

Advantages: Rapid development, Low coupling, Efficient debugging.

3. Monolithic vs. Microservices

The evolution of how we structure entire applications.

Aspect Monolithic Architecture Microservice Architecture
Structure Single, indivisible unit. All functions served in one place. Collection of smaller, independent units.
Database Shared database for the whole app. Each service often has its own database.
Communication Internal function calls. Lightweight mechanisms (HTTP API).
Pros Low cost, Simple to develop/deploy initially. Easy maintenance, High scalability, Low coupling.
Cons Tight coupling, Difficult to scale, Hard to maintain as it grows. Extra complexity, cross-cutting concerns, higher cost.

4. Specialized Architectures

Micro-Kernel (Plug-in) Architecture

Consists of a Core System (minimal functionality) and several Plug-in Modules (specialized processing).

  • Use Case: Applications with fixed core routines but dynamic rules that need frequent updates (e.g., Eclipse IDE, VS Code).
  • Requirement: Plugins need good "handshaking code" so the kernel recognizes them.

Event-Driven Architecture

Made up of decoupled, single-purpose event processing components. They receive and process events asynchronously.

  • Flow: Event → Event Queue → Event Mediator → Event Channel → Processors.
  • Pros: Agile, highly performant, easy debugging.
